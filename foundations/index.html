Brand Principles

Five non-negotiables that define what JioGames looks and feels like. Violating any one of these breaks the identity.

Dark-first
Every surface defaults to dark. Light is used for emphasis only — text, icons, CTA labels on green. No white or grey backgrounds ever.
Brand Principle
Green-first
--jio (#00A859) is the sole brand colour. Never use blue, purple, or indigo as an accent. All tiers use green — Ultimate uses a brighter green, not a different hue.
Brand Principle
Gaming DNA
Motion, depth, and cinematic feel over flat utility. Hero imagery, glow states, and spring animations communicate a premium interactive experience.
Brand Principle
Premium at every tier
Even free users experience premium UI quality. The design system does not downgrade for lower-tier users — monetisation is in content, not interface quality.
Brand Principle
Platform-native
Mobile, Web, and TV each get native feel — not lowest-common-denominator compromises. Touch, pointer, and D-pad each have first-class design.
Brand Principle

Product Principles

Six principles that govern product decisions — what to build, what to show, and in what order.

Speed over completeness
Show something fast. Defer secondary content, don't block the primary experience waiting for it. Skeletons over spinners.
Product Principle
Progressive disclosure
Show what's needed now. Offer depth on demand. Don't surface every option upfront — earn complexity by showing the user needs it.
Product Principle
Trust through consistency
The same component behaves identically in every context. Predictability is not boring — it is the foundation of trust.
Product Principle
Accessible by default
Accessibility is not a checklist appended at the end. It is a design constraint applied from the first sketch. Every component is built accessible.
Product Principle
Zero tolerance for jank
60fps minimum. Animation must add value, not distract. If an animation cannot be achieved at 60fps, it is removed — not reduced.
Product Principle
Mobile is the primary surface
Design mobile first, scale up to web and TV. Every layout decision starts at 390px and expands — never the reverse.
Product Principle

Platform Principles

Each platform has a different input model, viewing distance, and interaction contract. Design for all three — not a single lowest common denominator.

Mobile
  • Touch targets ≥ 44px at all times
  • Bottom navigation — never a sidebar
  • Swipe gestures supplement tap, never replace
  • Safe areas respected (notch, home indicator)
  • Thumb reach zone is the hero interaction zone
  • AppBar collapses on scroll — screen space is precious
Web
  • Hover states on all interactive elements
  • Full keyboard navigation — Tab, arrows, Enter, Escape
  • Right-click / context menus supported where appropriate
  • Desktop content density is acceptable — more per screen
  • Mouse precision allows smaller secondary targets (min 32px)
  • AppBar is always visible — sticky at top
TV
  • D-pad only — no mouse, no touch input
  • Focus ring is the sole affordance — never rely on hover
  • Minimum focusable target: 60px
  • Remote latency: never assume instant response
  • 10-foot rule: all text ≥ 24px, all icons ≥ 32px
  • No hover interactions — all states must be focus-driven

Token Philosophy

Tokens are the bridge between design decisions and code. The system uses three layers — primitive, semantic, and component — so that a single colour change propagates everywhere it is used.

Raw values. Never referenced directly in component CSS. They exist only to feed semantic tokens. Example: --_green-500: #00A859

Accessibility Principles

Accessibility is built in at the token and component level — not added after the fact. These four commitments apply to every component in the system.

WCAG 2.1 AA
Minimum target across all platforms and components. Contrast ratios, target sizes, and focus visibility are tested at every release.
Accessibility
Keyboard and remote
Every interaction is reachable by keyboard on web and by D-pad on TV. No interaction is pointer-exclusive.
Accessibility
Screen reader support
Semantic HTML, ARIA labels, live regions for dynamic content, and deliberate focus management on route transitions and sheet opens.
Accessibility
Reduced motion
All animations are gated on prefers-reduced-motion. The token system collapses durations to 0ms automatically — no per-component handling required.
Accessibility

Motion Principles

Animation communicates state and creates cinematic weight. It is never decorative. Every duration and easing is a token.

Purposeful
Every animation communicates a state change — appearing, dismissing, selecting, confirming. Remove any animation that doesn't tell the user something.
Motion Principle
Fast entry
Elements appear in 150–200ms (--dur-fast to --dur-default). Exit can be slower — 300–420ms. Never block the user waiting for an animation to finish.
Motion Principle
Spring for interaction
Use --spring or --spring-bounce curves for direct manipulation — drag, press, release, selection celebration. Linear easing is never used.
Motion Principle
Reduced motion respected
prefers-reduced-motion: reduce disables all non-essential animation automatically via tokens.css. No additional component code needed.
Motion Principle

Component Maturity Model

Every component carries a maturity level. Use only Level 3 (Ready) components in production. Levels 1–2 exist for design and engineering alignment — not user-facing code.

Level Name Meaning Badge
0 Idea Under discussion. Not yet designed or specified. —
1 Draft Design in progress. No implementation. Do not use in production. Draft
2 Beta Implemented and testable. Needs real usage to validate. API may change. Beta
3 Ready Stable. Validated in production. Safe to use across all surfaces. Ready
4 Deprecated Will be removed in the next major version. Migrate now. Deprecated

Non-Negotiable Rules

These are hard constraints — not guidelines. Every violation degrades the system's integrity.

Always use semantic tokens — never raw hex values in component or screen CSS.
Every interactive element must have a clearly visible focus state — outline: none without a replacement glow is a bug.
Every icon-only control must have an accessible label via aria-label or title.
Dark backgrounds only — no white or light surfaces except semantic states (error, positive feedback) with defined tokens.
Never use font weights outside 300 / 500 / 700 / 900. Weights 100, 200, 400, 600, 800 are banned.
Never use Inter, Outfit, Roboto, or any non-JioType font. JioType is the only allowed typeface.
