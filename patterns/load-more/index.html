Infinite Scroll, Pagination & Load More

Three distinct patterns for progressively revealing content. Each serves a different user mental model — choose based on whether users are browsing, searching, or managing a known collection.

Mobile Web TV

Purpose

Progressive loading patterns balance the cost of fetching all data upfront against the friction of manual navigation. The correct choice depends on whether users have a destination in mind (pagination), are browsing without one (infinite scroll), or want deliberate control over when more content loads (Load More). Getting this decision wrong creates poor experiences: infinite scroll on a library means users lose their place; pagination on a discovery feed adds unnecessary friction.

When to use / When not to use

Pattern Use when Do not use when JioGames examples
Infinite Scroll User is browsing without a specific destination. Content is roughly homogeneous (all games). Engagement is the goal — keep users in flow. User needs to return to an exact position. Content is search results. Back navigation must restore context. Footer links must be reachable. Home feed rails, genre browse, discovery pages
Pagination User needs to know their position in results. URL sharability matters (page=3 is linkable). User may navigate back to the same page repeatedly. Casual discovery feeds where position is irrelevant. TV surfaces where D-pad pagination is cumbersome. Search results, filtered catalogue views
Load More button User controls when to fetch. List has a known or moderate length. Page footer must be reachable. User may want to stop at a point in the list. Short lists where all items fit in one view. Infinite feeds where stopping points are arbitrary. Game library, purchase history, saved games

Anatomy

Infinite Scroll

  • Sentinel element — zero-height div at end of list; observed by IntersectionObserver
  • Skeleton row(s) — 2–4 skeleton cards shown while next batch loads
  • End-of-list indicator — "You've seen everything" when no more results
  • Error inline banner — with retry CTA, does not block existing content

Pagination

  • Previous / Next buttons — always visible when applicable; disabled at bounds
  • Page indicator — "Page 3 of 12" or numbered page chips on web
  • Results count — "Showing 41–60 of 240 results" pinned near controls
  • Navigation landmark — role="navigation" wrapping the controls

Load More

  • Load More button — secondary variant; includes count hint in label
  • Loading state — button enters spinner state; no layout shift
  • Result count label — "Showing 20 of 87" above the button
  • End state — button removed, "All items loaded" message shown

Flow

1
Initial load First batch of 20 items loads. Skeleton shown until first response arrives. IntersectionObserver (infinite scroll) or button (Load More) is mounted after first render.
2
Prefetch trigger Infinite scroll only: When the user is 5 items from the end of the current batch, silently fetch the next 20 in the background. This eliminates the loading gap in most connection conditions.
3
Load triggered Sentinel observed (infinite scroll) or button tapped (Load More). Skeleton appended to list. Button enters loading state. Pagination navigates to new page, full skeleton shown.
4
Content arrives Skeleton slots replaced with real items. Focus management: Load More — move focus to first new item. Pagination — move focus to first result item and announce new page via aria-live.
5a
End of results Show end-of-list state. Remove Load More button or sentinel. Pagination: disable Next button. Infinite scroll: show "You've seen it all" message.
5b
Load error Inline banner below existing content: "Couldn't load more — Retry". Existing items intact. Button reverts to non-loading state. aria-live="assertive" announces error.

Platform behavior

Mobile

  • Browse/home: infinite scroll. Store scroll position in history.state so back navigation restores position.
  • Game library: Load More button. Users managing a list want deliberate control.
  • Search: pagination with Prev/Next. Mobile pagination shows only Prev / Next + "Page N of M" — no numbered chips (too small to tap accurately).

Web

  • Search results: numbered page chips (1 2 3 … 12) + Prev/Next. URL param ?page=N is deep-linkable and shareable.
  • Browse feeds: infinite scroll with 20-item batches and prefetch at 5-from-end.
  • Game library: Load More button with "Showing N of M games" count label.
  • Keyboard: Load More button must be reachable by Tab; Enter triggers load.

TV

  • Infinite scroll is supported on TV — D-pad focus naturally moves into new items as they load.
  • When next batch loads during D-pad navigation: focus first new item automatically.
  • Avoid Load More button on TV — it interrupts the natural D-pad flow. Prefer automatic loading.
  • Numbered pagination is acceptable for search on TV; keep to 5 visible page chips max.

Accessibility

  • ·Load More button: aria-label="Load more games, 20 more available". Update the count as remaining items change.
  • ·Pagination nav: Wrap controls in <nav role="navigation" aria-label="Search results pages">. Current page: aria-current="page".
  • ·Infinite scroll announcements: Use aria-live="polite" region to announce count after each load, e.g. "Loaded 20 more games. Showing 60 total."
  • ·Focus recovery — Load More: After new items insert, move focus to the first newly added item so keyboard users can immediately navigate into new content.
  • ·Focus recovery — Pagination: After page change, move focus to the first result item (or the heading above results).
  • ·Scroll position persistence: Store scrollY in history.state before navigating away. Restore on back navigation to prevent user losing place in a long infinite scroll list.
  • ·Loading state: Apply aria-busy="true" to the list container while a batch is loading. Remove on completion.
