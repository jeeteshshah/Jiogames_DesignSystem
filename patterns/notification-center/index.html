Notification Center

A persistent, accumulated record of notifications the user can review and dismiss at their own pace — distinct from the transient Toast component, which delivers in-the-moment feedback.

Mobile Web TV

Purpose

Toast handles real-time, interruptive feedback that disappears after a few seconds. The Notification Center handles everything that needs to persist: pass expiry warnings, new game additions, achievements, failed payments. Users should be able to open the center at any time and review what happened — even if they were away from the app when events occurred. The center is the durable layer; Toast is the ephemeral layer. They serve different user needs and should not be conflated.

Toast — use for

  • · Action confirmation ("Game added to library")
  • · Copy success, form saved
  • · Transient errors ("Couldn't save")
  • · Auto-dismisses, no persistence needed

Notification Center — use for

  • · Pass expiry warnings
  • · New games added to pass
  • · Achievements, rewards earned
  • · Payment failures, billing events
  • · System messages (maintenance, updates)

When to use / When not to use

Use when

  • ·The event is relevant beyond the current session (pass expiry, payment issue)
  • ·The user may have missed a Toast (was away, screen off)
  • ·The notification requires a follow-up action (renew pass, retry payment)
  • ·The event is meaningful to log (achievement, reward)

Do not use when

  • ·The feedback is immediate confirmation of a just-completed action — use Toast instead
  • ·The message is purely informational with no action potential
  • ·The event repeats so frequently it would flood the center (e.g. every minor app update)
  • ·The message requires immediate interruption — use a modal or dialog instead

Anatomy

A
Page / panel header Title "Notifications" + "Mark all read" text action (right-aligned). Badge count shown in tab bar icon, not repeated in header.
B
Date group header Sticky label: "Today", "Yesterday", "Earlier this week", "Older". Muted text, small caps style. Groups help users scan without reading timestamps on every row.
C
Notification row Icon (type-colored) + title + body snippet + timestamp. Unread: green dot, bold title, slightly lighter surface (var(--surface-2)). Read: no dot, normal weight, darker surface (var(--surface-1)).
D
Unread indicator 8px circle in var(--jio), positioned left of icon column. Hidden when row is read.
E
Swipe-to-dismiss Swipe left on mobile reveals a red "Delete" action. Full swipe dismisses. Must have an accessible button fallback (long-press menu or row action icon).
F
Empty state Centered illustration area + "You're all caught up" heading + "New notifications about your games and pass will appear here" body copy. No CTA needed.

Flow

1
Event arrives Backend pushes notification. Badge count on tab bar icon increments. If app is foregrounded, a Toast may also fire for immediate awareness — these are complementary, not redundant.
2
User opens center Taps notification bell/tab. List loads grouped by date. Unread items visually distinct. "Mark all read" available if any unread exist.
3
Reading / marking Focusing a row marks it read (reduces unread count, removes dot, adjusts surface color). Accessible users: focus moves through list by Tab/D-pad, each item auto-marks as read on focus.
4
Tapping a notification Deep-links to the relevant screen (game detail, pass management, achievement screen). Row marked read. Back navigation returns to notification center.
5
Dismissal Swipe-to-delete or button removes row with brief collapse animation. After 30 days, notifications auto-expire server-side. User has no manual "clear all" bulk-delete — only per-item dismiss.

Platform behavior

Mobile

  • Full-screen page (standard navigation push). No overlay or panel.
  • Swipe-left on rows reveals delete action. Full horizontal swipe = instant dismiss.
  • Badge count on tab bar icon. Clears to zero when all items read.
  • Date group headers stick to top of scroll area as user scrolls through groups.

Web

  • Side panel (slide in from right, max-width 400px) or full-page route depending on viewport width.
  • Narrow (≤ 768px): full page. Wide: side panel with scrim behind.
  • No swipe-to-dismiss on web — use a delete icon button on each row instead.
  • Keyboard: Escape closes side panel. Tab navigates rows. Delete key on focused row = dismiss.

TV

  • Full-screen page. No panel — panels are awkward on TV focus model.
  • D-pad Up/Down navigates rows. Select/OK opens deep link. Back exits center.
  • Delete action: D-pad right on a focused row reveals "Dismiss" button, focusable with D-pad right again.
  • TV focus ring: var(--tv-focus-shadow) on each row.

Accessibility

  • ·Tab bar badge: aria-label="Notifications, 3 unread". Update the count dynamically. When zero: aria-label="Notifications".
  • ·Unread state: Do not rely on color dot alone. Unread rows should include a visually-hidden "Unread" label or use aria-label on the row that includes this context.
  • ·Mark read on focus: When a user tabs or D-pads to a row, mark it read. This matches sighted users' visual scan behaviour. Announce read status change via aria-live="polite".
  • ·Swipe-to-dismiss fallback: Every notification row must have a non-gesture dismiss path — a button with aria-label="Dismiss [notification title]". This serves keyboard, switch-access, and screen reader users.
  • ·Deep link rows: Each row should be a focusable element with role="link" or <a>. Include destination context in aria-label: "View Tomb Raider — new game added to pass".
  • ·Timestamps: Relative timestamps ("2 hours ago") should have a title attribute with the absolute datetime so users can hover/inspect for precision. Machine-readable: <time datetime="2026-06-29T10:30:00">.

Grouping and persistence rules

  • · Date groups: Today, Yesterday, Earlier this week, Older. Notifications older than 3 days collapse into "Older" group.
  • · Notifications persist until user dismisses them or they are 30 days old (auto-expire server-side).
  • · Do not duplicate identical notifications within 24 hours — deduplicate on backend before delivering.
