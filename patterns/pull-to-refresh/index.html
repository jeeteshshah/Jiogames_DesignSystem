Pull to Refresh

A touch gesture that lets mobile users reload content by pulling a scrollable surface downward past a threshold, triggering a refresh cycle.

Mobile

Purpose

Pull to Refresh gives users an immediately recognisable, low-friction way to fetch the latest content on mobile surfaces where the feed or list may change over time — such as the game home feed, notification list, or activity log. Because the gesture is native to mobile OS conventions (iOS and Android both implement it as a system pattern), users encounter no learning curve. The indicator provides continuous feedback during the pull distance, the commit threshold, and the async refresh cycle, so users always know the system is responding.

When to use / When not to use

Use when

  • ·The list or feed may have new content since last load (home feed, notifications)
  • ·The surface already scrolls vertically from the top
  • ·Refreshes are quick (< 5 seconds expected) — long waits erode trust in the gesture
  • ·Mobile only — users expect this gesture on touch devices

Do not use when

  • ·The list uses infinite scroll — loading older items at the bottom conflicts with pull gesture semantics
  • ·The list uses pagination — refresh would discard position and confuse users
  • ·Search results — user-defined queries don't benefit from background refresh; use an explicit refresh button
  • ·TV or Web — no touch input; use a visible refresh button or auto-polling instead
  • ·The screen has static content that does not change (settings, legal pages)

Anatomy

A
Scrollable container The full-screen scroll region that the user pulls down. Overflow behavior temporarily overridden to allow above-fold reveal.
B
Indicator track Reserved 64px zone above the first list item. Hidden at rest, revealed as user overscrolls. Centers the spinner horizontally.
C
Progress arc 24px SVG arc that fills proportionally as pull distance approaches the 80px commit threshold. Color: var(--jio). Stroke width: var(--spinner-stroke) (2px).
D
Spinner 24px circular spinner shown during the refreshing state. Continuous 360° rotation at var(--dur-icon-spin) (1000ms) linear. Replaces the progress arc.
E
Success checkmark Brief checkmark icon in var(--status-positive). Shown for exactly 400ms then fades with var(--dur-fast). Provides confirmation feedback before indicator disappears.

Flow

1
Idle List is scrolled to top. Indicator track is hidden. No visual change until overscroll begins.
2
Pulling (0–64px) Content follows finger with spring resistance factor 0.4 — the surface moves at 40% of finger velocity, providing tactile push-back. Indicator track reveals proportionally. Progress arc fills from 0%.
3
Pulling (64–80px) Indicator becomes fully visible at 64px. Arc continues filling toward 100%. A subtle haptic tick fires at 80px to confirm threshold hit (where available).
4
Releasing User lifts finger past 80px. Content snaps up with var(--dur-sheet) (400ms) spring easing to rest position showing indicator. Arc swaps to continuous spinner. Refresh request fires.
5
Refreshing Spinner rotates continuously. Existing content remains visible — no skeleton flash. New content is fetched in the background. List is not blanked during this state.
6a
Success Spinner swaps to checkmark (400ms dwell). List updates in place. aria-live="polite" region announces "Content refreshed". Indicator fades out with var(--dur-fast).
6b
Error Indicator shakes horizontally (ease-error easing, var(--dur-error)). Spinner color changes to var(--status-negative). aria-live announces "Refresh failed". Existing content remains. Indicator fades after 1.5s.

Platform behavior

Mobile

  • Full native gesture support. Use WebKit/Chromium overscroll-behavior or native SDK component.
  • Resistance factor 0.4 applied via JS touch handler if custom implementation needed.
  • Haptic confirmation at 80px threshold where device supports it.
  • Safe area aware — indicator stays below status bar.

Web

  • Not supported. Web users do not expect pull-to-refresh on desktop browsers.
  • Use a visible "Refresh" icon button in the page header or toolbar instead.
  • Auto-polling (e.g. every 30s) is acceptable for notification badges on web.

TV

  • Not supported. No touch input on TV remotes.
  • Add a focusable "Refresh" action in the page header, reachable by D-pad.
  • Consider background auto-refresh on a timer for content feeds.

Reduced motion

When prefers-reduced-motion: reduce is active: disable spring resistance animation (content tracks finger 1:1), replace spinning animation with a static arc, use instant show/hide for indicator (var(--dur-reduced-instant)). Error shake is replaced by a static red indicator with text label "Refresh failed".

Accessibility

  • ·Include a visually-hidden <div aria-live="polite" aria-atomic="true"> region in the DOM. On success, inject "Content refreshed". On error, inject "Refresh failed, please try again".
  • ·Always provide a visible refresh button (icon button in the app bar or page header) as a pointer/keyboard/switch-access alternative. The gesture alone is not sufficient.
  • ·During the refreshing state, set aria-busy="true" on the list container. Remove it when refresh completes or errors.
  • ·The spinner element should have role="progressbar" and aria-label="Refreshing". Hide it from the accessibility tree when not refreshing using aria-hidden="true".
  • ·Do not use color alone to distinguish success from error states — pair the color change with distinct iconography (checkmark vs. X) and the aria-live announcement.

Skeleton relationship

Pull to Refresh does not blank the list and replace with skeletons. The existing content stays visible throughout the refresh cycle. Skeletons are only appropriate on initial page load when no content exists yet. Replacing visible content with skeletons during a refresh creates unnecessary visual disruption and degrades perceived performance.
