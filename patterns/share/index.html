Share & Invite

Patterns for sharing game links, inviting friends, and distributing referral codes — using native APIs where available and graceful fallbacks where not.

Mobile Web TV

Purpose

Share and Invite surfaces allow users to spread discovery of games and bring friends into JioGames. There are three distinct sub-patterns: Share (copy/send a link to a game or piece of content), Invite (send a personalised invitation to a specific contact), and Referral (share a code that grants both parties a benefit). Each has different UX needs and platform constraints — particularly TV, where clipboard and contacts APIs are unavailable.

When to use / When not to use

Use when

  • ·Content is publicly reachable by the recipient (or you handle auth-gate resolution for the link)
  • ·The game detail page or content has a stable, shareable URL
  • ·Referral program is active and the user is eligible
  • ·Invite flow has a well-defined success state (invite sent, accepted)

Do not use when

  • ·The shared content is behind an auth gate and link resolution for unauthenticated recipients is not handled — the recipient would land on a login screen with no context
  • ·The content is user-generated and hasn't been reviewed — sharing could amplify harmful content
  • ·The referral program is not active — do not show referral UI for ineligible users

Anatomy

Share (link)

  • Share button — icon button (upload/share icon). Label "Share game".
  • Native share sheet — triggered via navigator.share() on mobile web and PWA.
  • Copy link fallback — bottom sheet with URL field + "Copy link" CTA, for environments where navigator.share is unavailable.
  • Copy feedback — button swaps to checkmark icon (var(--dur-instant)), reverts after 2s.

Invite friend

  • Input field — phone number or email. Label: "Invite a friend".
  • Contact picker — "Choose from contacts" button triggers native contacts API where permitted.
  • Send CTA — primary button, disabled until valid input.
  • Success state — inline success message below field. Toast "Invite sent" (positive variant) also fires.
  • Error state — inline field error. Toast "Couldn't send invite" (negative variant).

Referral code

  • Code pill — monospace text in a pill badge. Select-all on tap for easy manual copy.
  • Copy icon button — adjacent to pill. Swaps to checkmark on copy success.
  • Share button — shares pre-composed message including code and join link.
  • Benefit label — short description of reward for both referrer and referee.

Flow

1
User taps Share Share icon button tapped on game detail. Check for navigator.share support. If available on mobile, call it immediately with title, url, and optional text. No intermediate UI needed.
2
Native share (mobile) OS native share sheet appears. User selects app (WhatsApp, Messages, etc.). JioGames has no further control. On success: no Toast needed — the OS confirms. On user cancel: no action, no Toast.
3
Copy link fallback When navigator.share unavailable (desktop web, old browsers). Bottom sheet / popover slides up with URL field + "Copy link" button. User taps Copy: button swaps to checkmark, Toast fires "Link copied to clipboard", reverts after 2s.
4
Invite friend User enters phone/email or selects contact. Taps Send. Button enters loading state. On success: inline "Invite sent!" message + positive Toast. On failure: inline field error + negative Toast.
5
Referral share User taps code pill (selects all) or copy icon. Toast "Referral code copied". Share button opens native share with pre-composed message: "Join JioGames with my code [CODE] and get [BENEFIT]: [link]".

Platform behavior

Mobile

  • Use navigator.share() for all share actions — shows native OS share sheet.
  • Invite: contact picker via navigator.contacts.select() where supported (Android). Fallback: manual input field.
  • Share surface: bottom sheet for copy-link fallback, slides up from bottom edge.
  • Referral pill: tap selects all text. Copy icon triggers clipboard write.

Web

  • Check navigator.share support at click time — available in Chrome/Edge, not Firefox/Safari desktop.
  • Fallback: dropdown/popover anchored to the Share button. Shows URL + Copy link + optional direct-share icons (WhatsApp, Twitter links).
  • Invite: no contacts API on web — input field only.
  • Escape key closes popover. Focus returns to trigger button on close.

TV

  • No clipboard API, no contacts API, no native share sheet on TV.
  • Share: show a full-screen dialog with a QR code linking to the game. "Scan with your phone to share or play on mobile."
  • Invite: not available on TV. Share button leads to QR code only.
  • Referral: show code on screen in large, readable text alongside QR code. User reads/copies on their phone.

Accessibility

  • ·Share button: aria-label="Share game" (icon-only buttons must have an accessible name). If button label is visible text, no aria-label needed.
  • ·Copy link: On successful copy, announce via aria-live="polite": "Link copied to clipboard". This supplements the Toast for screen reader users who may not perceive the Toast visual.
  • ·Referral code field: role="textbox" aria-label="Your referral code" aria-readonly="true". Include tabindex="0" so keyboard users can focus and copy via Ctrl+C / Cmd+C.
  • ·Copy button state change: When icon swaps to checkmark, update aria-label to "Link copied" so screen readers announce the state change. Revert label when icon reverts.
  • ·Invite input error: Error message linked to input via aria-describedby. Error announced via aria-live="assertive" because it's an action failure requiring immediate attention.
  • ·Bottom sheet / popover focus: On open, focus moves to first interactive element inside. On close, focus returns to the trigger Share button.
  • ·TV QR dialog: QR code image must have alt="QR code — scan to share [Game Title]". The URL should also be displayed as readable text below the QR code for users who cannot scan.
